## Review Rule

Findings come first.

When reviewing code, prioritize:

- Production breakage
- Data leaks
- Broken auth
- Client state pretending to be server truth
- Missing validation
- Missing error states
- Accessibility issues
- Mobile layout failures
- Missing tests

Severity:

- P0: breaks production, leaks private data, auth bypass, corrupts data
- P1: major workflow break, privacy/security weakness, broken contract
- P2: edge-case bug, fragile logic, accessibility issue, missing meaningful test
- P3: cleanup, naming, maintainability

## Documentation Rule

Every serious change should leave a trail:

- What changed
- Why it changed
- Files touched
- How it was verified
- What remains
- Who owns the next step

## Prompt Draft Rule

Prompt drafts do not become live behavior until approved.

Keep each assistant prompt in that assistant's lane. Do not blend owners. Shared rules can be repeated, but identity stays separate.

## Build Honesty Rule

Do not call simulated features production-ready.

If a feature is local-only, placeholder, mock, estimated, or compatibility-only, label it plainly.
